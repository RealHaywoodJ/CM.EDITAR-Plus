OBJECTIVE
Build a Windows‑first desktop application (with cross‑platform UI portability in mind) that:
• Discovers every live New (ShellNew) entry on the machine (HKCU-first, HKCR fallback read-only).
• Lets users inspect metadata and risk for each entry.
• Lets users stage edits (enable/disable, change FileName/Command/Data/NullFile, change display name) with a dry‑run preview.
• Applies changes safely (HKCU only at runtime) with snapshot export, atomic apply, SHChangeNotify refresh, verification, audit log, and automatic rollback on failure.
• Installs an optional per‑user New+ submenu that appears immediately after the default New and hosts curated templates (static and dynamic).
• Provides a Template Manager and a non‑elevated FileCreator runtime for dynamic templates via secure IPC.

CONSTRAINTS (MANDATORY)
1. Runtime writes must be HKCU only. No HKLM/HKCR writes at runtime without explicit elevated installer action.
2. Always snapshot before any registry write. Snapshot format: export .reg to %LocalAppData%\CM.EDITAR\Backups\<ISO-timestamp>.reg and write metadata JSON with timestamp, user SID, list of keys, and SHA256 of the .reg file. Never overwrite snapshots.
3. Dry‑run preview required: show exact registry keys and values to be added/modified/deleted before apply.
4. Explicit user confirmation required before apply. For high‑risk actions require typed confirmation phrase.
5. Automatic rollback: if verification fails after apply, import snapshot and log the event.
6. Templates are data only by default. Command templates must require explicit user approval and show security warnings.
7. IPC must be authenticated: per‑install secret stored with DPAPI on Windows; named pipe ACL restricted to current user.
8. Stop and request human approval before any destructive action (deletions, HKLM writes, or mass changes).

TARGET PLATFORM & TECH STACK (RECOMMENDED)

CORE TECHNICAL REQUIREMENTS (DETAILED)
A. Smart Registry Logic (must handle Windows shell nuances)
1. Discovery: enumerate ShellNew entries by scanning HKCU\Software\Classes for keys with a ShellNew subkey. For discovery only, read HKCR fallback. Output canonical JSON: { extension, displayName, shellNewType, fileName, command, data, visible, sourceKey, progIdResolved, risk }.
2. ProgID resolution: for a target extension, resolve the effective ProgID using:
   • Check HKCU\Software\Classes\<.ext>\(Default) for user override.
   • If absent, check HKCR\<.ext>\(Default).
   • If the extension has a UserChoice/ProgId redirection (UserChoice key), detect and handle it appropriately (do not overwrite UserChoice; write to the resolved ProgID path).
   • If a ProgID exists, write ShellNew under HKCU\Software\Classes\<ProgID>\ShellNew (not directly under .ext) to ensure Explorer honors it.
3. ItemName and NullFile: when creating ShellNew entries, include NullFile (REG_SZ or REG_NONE as appropriate) and ItemName (localized display name) to force display in the New menu. Provide localized ItemName fallback to the extension display name.
4. Extensionless/Blank file support: implement logic to add a "New (blank file)" entry (extensionless) by creating the correct ShellNew registration (e.g., using the * or . key or a special ProgID) and ensure it appears in modern Windows 11 New menu. Provide safe defaults and require explicit confirmation.
5. Windows 11 modern menu handling: ensure the New item appears in the modern context menu (not only in "Show more options"). Use the correct ShellNew registration and call any required shell ordering keys or shell folder registration techniques to ensure visibility.

B. Apply & Refresh
1. Preflight: enumerate exact registry keys/values to be changed and present a human‑readable diff (add/modify/delete) in the UI.
2. Snapshot: export the exact keys to a .reg file in Backups and write metadata JSON with SHA256.
3. Apply: write only to HKCU via a RegistryService abstraction. After write, call SHChangeNotify (or appropriate shell APIs) to refresh Explorer UI immediately.
4. Verify: re‑discover the New menu and confirm the expected entry is visible. If verification fails, automatically restore snapshot and report failure.
5. Audit: append a signed changelog entry to %LocalAppData%\CM.EDITAR\Audit\changes.log with timestamp, user SID, operation summary, and snapshot path.

C. New+ Submenu
1. Per‑user parent key: register a per‑user submenu labeled "New+" under the same parent as the default New so it appears immediately after default New. Guarantee ordering by using numeric ordering keys or explicit shell ordering technique.
2. Entries: support ShellNew\FileName for static templates and ShellNew\Command for dynamic templates that call CM.EDITAR.FileCreator.exe with args or via named pipe.
3. Installer option: provide in‑app and MSI installer flows to register New+ per‑user. Portable ZIP must register per‑user entries without elevation and document limitations.

D. Template Manager & FileCreator
1. Template model: id (UUID), name, extensions[], category, defaultFilename, templateType (NullFile/FileName/Command/Data), templateSource, placeholders[], createdAt, modifiedAt, author.
2. Storage: static templates stored at %AppData%\CM.EDITAR\Templates\<id>\template.* and metadata.json.
3. Template Manager UI: list, create, edit, preview, placeholder editor, map templates to extensions, import/export packs.
4. FileCreator CLI: CM.EDITAR.FileCreator.exe --create --template-id <id> --target "%1" that resolves placeholders and writes files atomically (write to temp file then atomic rename).
5. IPC: named pipe server with ACL restricted to current user; authenticate with per‑install secret stored via DPAPI. FileCreator validates template id and refuses unknown ids.

E. Security & Safety
1. Template sanitization: static templates safe; Command templates require explicit user approval and show a security warning. Disallow network paths or auto‑run behavior in templates.
2. IPC auth: per‑install secret stored with DPAPI; named pipe ACL limited to current user; token validation on each request.
3. Least privilege: runtime components non‑elevated; installer elevated only for install/uninstall tasks.
4. Logging: detailed logs in %LocalAppData%\CM.EDITAR\Logs with opt‑in telemetry only if user consents.
